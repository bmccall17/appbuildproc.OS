# bmccall17.github.io / darketype Development Dossier

## Original Problem And Audience

**Observed:** The repository begins as a GitHub Pages portfolio and quickly becomes darketype: manifesto, weblog, visual experiments, ship log, success/career microsites, publication workflows, and reflections on other projects.

**Inferred:** The audience is public: collaborators, employers, readers, and the builder himself as a reflective operator.

## Earliest Prototype

**Observed:** GitHub Pages deployment appears in the second commit. The first day includes personal details, manifesto, ship v0.1.0, first entry, and v0.1.1 loop engagement.

## Major Milestones

- **Deployment:** GitHub Actions for GitHub Pages within minutes of initial commit.
- **Identity:** darketype manifesto and glitch visual language.
- **Weblog:** First entry and later dynamic weblog engine.
- **Visual systems:** Heatmap, pixel chaos, singularity, CSLP, logo factory, responsive matrix.
- **Release process:** SHIP_LOG and deploy guide.
- **Career/product surfaces:** `/success`, bookable/call links, resume/card updates.
- **Cross-project reflection:** Posts for repOptics, agent828, TarotTALKS, trust registry, backlog, and process lessons.
- **Publishing automation:** Weblog build, OG generation, Hashnode sync, generated cover/SEO support.

## Deployment History

**Observed:** GitHub Pages workflow is added on day one. Later generated rebuild commits and static workflow keep the site current.

## Architectural Decisions

**Observed:** No ADR trail was inspected. Process lives in `.agent` workflows, ship log, deploy guide, package scripts, and the content pipeline structure.

## Admin And Operations Evolution

**Observed:** Instead of admin panels, the repo develops publishing operations: entry workflow, orchestrator, self-healing workflow, Hashnode sync, quarantine, generated indexes, and OG image generation.

## Testing And Reliability Evolution

**Observed:** Package scripts include `test`; workflows rebuild and sync contributions. The reliability concern is publication integrity and rebuild correctness rather than application data safety.

## Presentation And Distribution Evolution

**Observed:** Presentation is the product. Visual identity, responsive layout, OG/favicons, Hashnode, weblog index, and project cards are central.

## Major Direction Changes

- Portfolio -> manifesto/weblog.
- Static pages -> dynamic generated weblog.
- Personal site -> cross-project publication/reflection surface.
- Manual posts -> build/OG/Hashnode pipeline.

## Current State

**Observed:** darketype is a public meta-process surface: it publishes, reflects, syndicates, and packages stories from the rest of the corpus.

## Process Lessons

**Inferred:** darketype shows that Brett externalizes process learning by turning project history into public artifacts. The application-building process includes a reflection/publication loop, not only code delivery.
